# Security Policy

Security reports are welcome and should be handled privately.

## Supported Scope

Reports are especially useful for:

- authentication and authorization issues
- privilege escalation
- credential exposure
- API token handling
- session, TOTP or WebAuthn weaknesses
- CSRF, XSS, SQL injection, command injection or path traversal
- backup repository access control issues
- secret storage or secret broker weaknesses
- restore or download access bypasses
- sensitive information disclosure

## Reporting A Vulnerability

Do not open a public issue for a vulnerability.

Use a private maintainer channel. If the repository host provides private security reporting, prefer that path. If no dedicated mailbox is published, contact the project owner through the repository host profile or another private channel.

## What To Include

- affected commit or version
- impact summary
- reproduction steps
- proof of concept, if safe to share
- relevant logs with secrets redacted
- suggested remediation, if available

## Disclosure Expectations

Please:

- give maintainers reasonable time to investigate and fix the issue
- avoid public disclosure before a fix or mitigation is available
- avoid destructive testing
- avoid accessing data that is not yours

## Response Goals

Maintainers aim to acknowledge, validate and coordinate fixes for credible reports. No specific SLA is guaranteed at this stage of the project.
