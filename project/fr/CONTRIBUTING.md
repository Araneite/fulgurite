## Contributions Attendues

Les contributions les plus utiles sont :

- corrections de bugs reproductibles
- améliorations ciblées de documentation
- tests pour des comportements existants
- améliorations d’accessibilité et d’ergonomie
- durcissements de sécurité discutés avec les mainteneurs
- petits travaux alignés avec [ROADMAP.md](ROADMAP.md)

Les gros changements d’architecture, sujets de licence, limites commerciales, plugins et packaging doivent être discutés avant implémentation.

## Workflow

1. Ouvrir une issue pour les sujets larges ou ambigus.
2. Créer une branche ciblée.
3. Garder le changement aussi petit que possible.
4. Mettre à jour la documentation si le comportement, l’API ou le setup changent.
5. Ajouter ou mettre à jour les tests lorsque nécessaire.
6. Inclure des notes de vérification dans la pull request.
7. Confirmer l’acceptation de [CONTRIBUTOR_TERMS.md](CONTRIBUTOR_TERMS.md).

Texte suggéré dans la pull request :

```text
I have read and agree to CONTRIBUTOR_TERMS.md for this contribution.
```

## Attentes Techniques

- Respecter la structure et le style existants.
- Écrire les commentaires de code source en anglais.
- Éviter les refactors sans lien direct.
- Ne pas committer de secrets, bases runtime, `.env` locaux ou dumps de production.
- Mettre à jour les pages de documentation concernées.

## Propriété Des Contributions

Vous conservez la propriété du contenu original soumis. Vous accordez au projet les droits nécessaires pour relire, fusionner, maintenir et distribuer la contribution dans le dépôt public source-available.

Les fonctionnalités écrites par un contributeur ne seront pas déplacées dans une fonctionnalité payante ou commerciale sans accord écrit préalable.
